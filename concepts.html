HTTP (HyperText Transfer Protocol)

HTTP is the set of rules that browsers and web servers use to talk to each other. When you visit a website, your browser sends an HTTP request and the server sends back an HTTP response. That back-and-forth is happening constantly in the background every time you use the web.

Every HTTP message has a few parts: a start line (describing what is being asked or what happened), headers (extra info like browser type or file format), and optionally a body (the actual content being sent or received).

One important thing about HTTP is that it is stateless, meaning each request is completely separate. The server does not automatically remember who you are from one request to the next. That is why websites use things like cookies and sessions to keep track of logged-in users. HTTP/1.1 added the ability to reuse connections. HTTP/2 added multiplexing so multiple requests could run over one connection. HTTP/3 switched away from TCP entirely to fix some deeper performance issues.

HTTP Request Methods

Every HTTP request uses a method that tells the server what kind of action to take:

  • GET: Load a resource. This is what happens every time you open a page or click a link.
  • POST: Send data to the server, like submitting a sign-up form.
  • PUT: Replace a resource on the server with new data.
  • PATCH: Update just part of a resource instead of replacing the whole thing.
  • DELETE: Remove a resource from the server.

These five methods are also the building blocks of REST APIs, which is how most modern web apps and mobile apps communicate with their backends.

HTTP Status Codes

Every server response includes a status code that tells you whether the request worked or not. They are grouped into five ranges:

  • 1xx (Informational): The request is still being processed. You rarely see these directly.
  • 2xx (Success): It worked. The most common is 200 OK.
  • 3xx (Redirect): The resource moved. 301 Moved Permanently sends the browser to a new URL.
  • 4xx (Client Error): Something was wrong with the request. 404 Not Found and 403 Forbidden are the ones most people recognize.
  • 5xx (Server Error): The server had a problem. 500 Internal Server Error is the most generic one.

HTTPS and Encryption (SSL/TLS)

HTTPS is just HTTP with encryption added. Without it, everything sent between your browser and a server goes as plain text, which means anyone watching the network traffic could read your passwords or credit card numbers. HTTPS wraps everything in a layer of encryption so that cannot happen.

The encryption layer is called TLS (Transport Layer Security). You might also see the older name SSL, which TLS replaced. When you visit an HTTPS site, your browser and the server do a quick TLS handshake where they agree on encryption keys. After that everything sent between them is scrambled. Even if someone intercepts it, they just see garbage without the key.

The handshake adds a little overhead (roughly 110ms vs 50ms without it), but modern optimizations like session resumption cut that down by reusing key info from a previous connection. Most browsers now show a warning if a site is not using HTTPS, and Google search gives HTTPS sites a small ranking boost.

Caching

Caching means storing a copy of something so you do not have to go get it again every single time someone asks for it. It happens at a few different levels:

  • Browser cache: Your browser saves copies of images, CSS, and scripts locally so they load instantly on repeat visits.
  • Server cache: The web server stores the result of expensive database queries or page renders so it does not have to redo them constantly.
  • CDN cache: A CDN stores copies of your content at edge servers around the world so users nearby get it quickly.
  • Reverse proxy cache: A proxy server like NGINX can cache responses from the backend and serve them directly.

HTTP lets servers control caching through headers like Cache-Control: max-age=3600, which tells browsers how long to keep a copy before checking for an update. Good caching can make a huge difference in how fast a site feels.

Load Balancing

Load balancing is the practice of splitting incoming traffic across multiple servers so no single machine gets overloaded. A load balancer sits in front of the server pool and decides where to send each request based on different rules:

  • Round-robin: Sends requests to servers in rotation, one after another.
  • Least-connections: Picks whichever server currently has the fewest active connections.
  • Weighted: Sends more traffic to faster or more powerful servers in the pool.

Load balancers also do health checks. If a server stops responding, the balancer automatically removes it from rotation and splits its traffic among the remaining healthy servers. This is how big platforms stay up even when individual machines fail.

Reverse Proxy

A reverse proxy is a server that sits in front of your real backend servers and handles all incoming traffic first. From a user's perspective it just looks like your website. Behind the scenes it is doing things like hiding your server's real IP address, handling SSL encryption, caching common files, and routing requests to the right backend service.

NGINX is probably the most popular reverse proxy. Cloudflare's entire global network also works as a reverse proxy at a huge scale, which is part of why it can absorb DDoS attacks without taking your site down.

Content Delivery Networks (CDNs)

A CDN is a worldwide network of servers that cache your content and serve it from wherever is closest to the person requesting it. Without a CDN every request has to travel to your main server, no matter how far away the user is. With a CDN a user in London hitting a site hosted in Dallas might actually get files from a CDN server in London, cutting the load time significantly.

CDNs are most useful for static content like images, videos, fonts, and CSS files that can be safely cached. CDNs also absorb a lot of traffic away from your main servers, which makes DDoS attacks much harder to pull off. Cloudflare, Akamai, and Amazon CloudFront are the biggest ones. Cloudflare alone operates in over 300 cities worldwide.

Scalability

Scalability is about how well a system handles growth. There are two basic ways to scale a web server setup:

  • Vertical scaling: Making the server more powerful by adding more CPU, RAM, or faster storage. Simple but there is a ceiling to how far you can go, and if that one machine fails everything goes down.
  • Horizontal scaling: Adding more servers and splitting traffic across all of them with a load balancer. You can keep adding machines as needed, and losing one does not take the whole thing down.

Cloud platforms make horizontal scaling really easy. AWS and Google Cloud can automatically add new instances when traffic spikes and remove them when things calm down. This is called auto-scaling and it is how sites like Netflix handle millions of users without manually managing servers.

Virtual Hosting

Virtual hosting lets a single server host multiple completely different websites at the same time. When a request comes in, the server reads the Host header to figure out which site is being asked for and serves the right content. This is how shared hosting works: one physical server might be running thousands of websites, each with their own domain and files.

Both Apache and NGINX support virtual hosting natively. Without it, every website on the internet would need its own dedicated server, which obviously would not work.

DNS (Domain Name System)

DNS is what translates a human-readable domain name like "google.com" into an IP address like "142.250.80.46" that computers can actually route to. When you type a URL, your computer first does a DNS lookup to find the IP address, then connects to that address to make the HTTP request.

DNS is also used in load balancing. A domain can point to multiple IP addresses, and DNS can hand out different ones to different users, effectively spreading the load. This is called DNS-based load balancing and is one of the simpler ways to distribute traffic.

Apache HTTP Server

Apache is the open-source web server that dominated the internet for about 20 years after its release in 1995. It uses a modular design where features can be turned on or off through plug-in modules, which made it very flexible and customizable. Apache also supports .htaccess files that let you configure settings on a per-directory basis without restarting the server, which is why shared hosting providers love it. As of April 2026 it holds about 23.7% of the web server market.

NGINX

NGINX (pronounced "engine-x") was created by Igor Sysoev in 2004 specifically to solve the problem of handling huge numbers of simultaneous connections. Its event-driven design lets a small set of worker processes each handle thousands of connections without the memory overhead that Apache's per-connection process model required. Today it is used both as a standalone web server and as a reverse proxy and load balancer sitting in front of other servers. It is currently the most popular web server in the world at around 32.7% market share.
