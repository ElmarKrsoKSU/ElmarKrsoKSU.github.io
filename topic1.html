Where It All Started (1989-1991)

The whole thing kicked off in March 1989 when Tim Berners-Lee, a British computer scientist working at CERN in Switzerland, wrote up a proposal called HyperText and CERN. The basic idea was to make it easier for scientists to share research without having to email files back and forth. His manager scribbled "Vague but exciting" on the cover page, but approved it anyway.

By Christmas 1990, Berners-Lee had the first web server up and running on a NeXT workstation. The machine only had a 2 GB hard drive and a 256 MHz CPU, which sounds almost laughable now. He literally taped a note to the front of it that said "This machine is a server. Do not power it down!" That computer is still at CERN today. The server software was called CERN HTTPd, and he also built the first browser alongside it, which he named "WorldWideWeb."

Things spread fast. In December 1991 the first web server outside of Europe went online at the Stanford Linear Accelerator Center in California. By late 1992 people were already keeping a public list of servers because there were enough of them to track.

The Web Explodes (1993-1994)

In 1993 CERN did something that changed everything: they put all the web technology into the public domain so anyone could use it for free. After that the numbers went crazy. There were over 500 web servers by the end of 1993, and just twelve months later that had jumped to over 10,000 servers and 10 million users. The web went from a research project to a global thing in basically four years.

Also in 1993, the National Center for Supercomputing Applications (NCSA) at the University of Illinois released NCSA HTTPd, the second big web server after Berners-Lee's original. It could serve regular HTML files but also run programs and send back the output, thanks to a new invention from that same year: CGI (Common Gateway Interface), created by Rob McCool.

CGI was a big deal because it meant a web page could actually respond to what a user did. Instead of just showing a fixed file, the server could run a program, maybe pull from a database, and send back something different every time. That is basically how web applications started. NCSA HTTPd also worked with the Mosaic browser, which was the first browser that could show images inline in a page.

Apache Takes Over (1995)

When Rob McCool left NCSA in mid-1994, nobody was maintaining NCSA HTTPd anymore. A bunch of webmasters who depended on it started writing their own fixes and improvements. One of them was Brian Behlendorf, who ran the website for Wired Magazine (HotWired). He and a developer named Cliff Skolnick set up a mailing list so everyone could share their patches.

By February 1995, eight people had joined and started working together as what they called the Apache Group. They took NCSA HTTPd 1.3 as a starting point, applied all the patches they had collected, tested it on their own servers, and put out the first official release of the Apache HTTP Server in April 1995.

The name "Apache" supposedly has two stories. The official one from the Apache Software Foundation is that it was named after the Apache Native American tribe to honor their endurance. The other story, which was actually on their website for years, is that it was just "a patchy server" because of all the patches it was made from. Both explanations got passed around and neither one was ever really settled.

Apache 1.0 came out December 1, 1995. By April 1996 it had already passed NCSA HTTPd to become the number one web server. It stayed there for almost 20 years. By 2009 it was running over 100 million websites, which was a first for any web server.

When Apache Started Struggling (Late 1990s)

During the dot-com boom, Apache started hitting a wall. The way it worked was that every new connection to the server got its own process or thread. For a small site that was fine, but once you had tens of thousands of people connecting at the same time, the server would run out of memory and slow to a crawl.

Engineers started calling this the C10K problem, meaning the challenge of handling 10,000 concurrent connections on one machine. A software engineer named Dan Kegel wrote a paper about it in 1999 that became pretty well-known. The basic conclusion was that the architecture itself needed to change, not just the settings.

NGINX Changes the Game (2004)

The fix came from Igor Sysoev, a Russian systems administrator at Rambler, which was one of Russia's biggest web portals. Rambler was already handling hundreds of millions of page views a month in the early 2000s, so Sysoev knew firsthand how bad the scaling problem was. He tried tweaking Apache first but figured out pretty quickly that the whole architecture needed to be redesigned.

Starting in 2002, he built a new server from the ground up using an event-driven approach. Instead of one process per connection, a small set of worker processes could each handle thousands of connections at once without blocking. The memory usage was way lower and performance under heavy load was much better.

NGINX went public on October 4, 2004. It caught on fast in Russia first, then spread globally as other developers dealing with the same scaling issues tried it out. By September 2008, Rambler was using it to handle 500 million requests per day. As of April 2026, NGINX is the most used web server in the world at about 32.7% of all websites.

Microsoft Enters the Market (1996)

While all the open-source stuff was happening, Microsoft released Internet Information Services (IIS) bundled with Windows NT in 1996. It was built for the Windows ecosystem, so companies already running Windows infrastructure tended to use it since it all worked together out of the box.

Through the late 1990s the server market basically split three ways: Apache for Linux/Unix servers, IIS for Windows shops, and Netscape Enterprise Server for companies that wanted commercial support. The competition between them pushed all three forward pretty fast and helped nail down a lot of the web standards we still use today.

What This All Means Today

Every phase of web server history built on the one before it. Static HTML pages led to CGI and dynamic content. CGI-based servers led to modular open-source servers like Apache. Apache's limitations led to event-driven servers like NGINX. And all of that laid the groundwork for the cloud, containers, and microservices that we cover in the next section.
